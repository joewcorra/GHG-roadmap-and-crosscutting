GitHub Repository Architecture

Organizational Structure

We will use a modular repository structure with separate repos for each major GHG source category, coordinated through an overarching GitHub Project board.

Repository Structure

github.com/your-org/
├── ghg-inventory-project/ # GitHub Project (Kanban board)
├── ghg-energy/ # IPCC Category 1: Energy
├── ghg-ippu/ # IPCC Category 2: Industrial Processes
├── ghg-agriculture/ # IPCC Category 3: Agriculture
├── ghg-lulucf/ # IPCC Category 4: Land Use
├── ghg-waste/ # IPCC Category 5: Waste
├── ghg-core/ # Shared utilities & documentation
└── syrinx/ # Internal data harmonization package

Rationale for Multi-Repo Approach

Advantages: - Sector independence: Teams can work without conflicts - Clear ownership: Each sector lead owns their repository - Focused permissions: Control access by sector - Separate release cycles: Update sectors independently - Smaller codebases: Easier to understand and maintain - Specific issues/PRs: Sector-specific discussions stay organized

Coordination via GitHub Project: - Cross-repo task tracking - Unified timeline and milestones - Dependencies between sectors visible - Overall inventory status dashboard

Future Decision Points

Phase 1 Decisions (Months 0-6)

1.1 Workflow Orchestration: {targets} vs Scripts

Decision: Whether to implement {targets} pipeline management

Current Approach: Sequential R scripts (01_download.R, 02_clean.R, etc.)

Consider {targets} when: - Scripts become complex with many interdependencies - Rebuilding everything is taking too long - Need fine-grained caching of intermediate results - Multiple people running overlapping analyses

Evaluation Criteria: - Is manual script execution becoming error-prone? - Are we re-running expensive calculations unnecessarily? - Would the learning curve justify the benefits?

Timeline: Re-evaluate after MVP (Month 4)

1.2 Testing Strategy: Depth vs Speed

Decision: How comprehensive should automated testing be?

Current Approach: Basic unit tests for calculation functions

Options: - Minimal: Test core calculations only - Moderate: Add data validation tests - Comprehensive: Include integration tests, benchmark tests

Consider expansion when: - Bugs are discovered in production - Onboarding new team members who break things - Preparing for external review

Timeline: Decide by Month 3 based on bug frequency

Phase 2 Decisions (Months 6-12)

2.1 Database Implementation: DuckDB

Decision: When/whether to migrate from CSVs to DuckDB

Current Approach: Well-organized CSV files

Triggers for migration: - ✓ Query performance becomes problematic (joins taking >1 minute) - ✓ File management becomes unwieldy (>100 CSV files) - ✓ Need for referential integrity is critical - ✓ Team size grows (>5 active users) - ✓ Complex analytical queries are frequent

Migration Path:

# Phase 1: Read CSVs (current)
data <- read_csv("data/activity/electricity.csv")

# Phase 2: Add DuckDB option
con <- dbConnect(duckdb::duckdb(), "ghg_inventory.duckdb")
data <- dbReadTable(con, "electricity")

# Functions remain unchanged
calculate_emissions(data, emission_factors)

Implementation Steps: 1. Pilot DuckDB with one sector 2. Benchmark performance vs CSVs 3. Evaluate team comfort level 4. Decide to expand or revert

Timeline: Evaluate Month 8, implement Month 9-10 if approved

Phase 3+ Decisions (Year 2+)

3.1 Advanced Uncertainty Quantification

Decision: Implement Monte Carlo simulation

Current Approach: Use published uncertainty ranges from EPA

Enhanced Approach: - Probability distributions for emission factors - Correlation structures between parameters - Propagation through calculations - Full uncertainty analysis per IPCC guidelines

Package: {mc2d} or {propagate}

Triggers: - Required for peer review or publication - IPCC reporting requirements - Stakeholder requests for uncertainty bounds

Decision-Making Framework

When to Add Complexity

Ask these questions:

  1. Does the current simple approach actually cause pain?
    • If no → Don’t add complexity
    • If yes → Continue evaluation
  2. What’s the cost of the status quo?
    • Time wasted?
    • Errors introduced?
    • Opportunities missed?
  3. What’s the cost of the new approach?
    • Learning curve?
    • Maintenance burden?
    • Technical debt?
  4. Who benefits and how much?
    • Entire team or one person?
    • Daily impact or occasional?
    • Critical or nice-to-have?
  5. Is it reversible?
    • Can we go back if it doesn’t work?
    • What’s locked in?
  6. Do we have capacity?
    • Time to implement properly?
    • Skills on the team?
    • Bandwidth to maintain?

Decision Authority

Tech decisions by data system developer: - R package choices - Code organization - Git workflow details - Development tools

Collaborative decisions: - Database implementation - Major workflow changes - Data access/sharing policies - Public-facing features

Sector lead decisions: - Sector-specific methodologies - Data source selections - QA/QC thresholds - Reporting priorities

Review Milestones

Month 3 Review: - Is CSV approach working? - Is syrinx package valuable? - Are GitHub repos effective? - Need to adjust course?

Month 6 Review: - Ready for DuckDB evaluation? - Need centralized data hub? - Testing strategy adequate? - Workflow orchestration needed?

Month 12 Review: - System meeting goals? - What future capabilities to prioritize? - Resource needs for Year 2? - External engagement strategy?
